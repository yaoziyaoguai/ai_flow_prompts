# Architecture Review —— 架构专项审计

## 适用场景

- 新增一个模块或服务前的架构设计评审
- 对现有架构做专项健康检查
- 重大重构前评估架构方向
- 技术选型决策中的结构性影响评估

## 不适合场景

- 小改动的 code review（不需要架构级别评审）
- 已经充分讨论并确定的架构
- 紧急修复（先修，事后再评审）
- 需要综合审计代码、测试、文档、安全和流程（用 `coding-agent/independent-audit.md`）

## 可组合模块

- `common/context-brief.md` —— 理解当前架构
- `common/brutal-honesty.md` —— 诚实的架构评估
- `common/red-team.md` —— 攻击架构的薄弱环节
- `snippets/architecture-quality.md` —— 架构质量标准

## 完整模板

---

# Architecture Review

对以下架构设计进行专项审计。只关注结构、边界、依赖和演进风险；不要扩展成综合代码审计。

如果需要综合审计代码、架构、测试、文档、安全和流程，请改用 `coding-agent/independent-audit.md`。本模板可以作为综合审计中的架构专项模块。

## 评审维度

### 1. 模块边界
- 模块之间的职责是否清晰？
- 是否存在边界模糊的"万能模块"？
- 模块间的依赖关系是否合理（单向依赖优于双向）？
- 是否存在循环依赖？

### 2. 抽象层次
- 抽象层次是否一致？（不要在同一模块中混用高层抽象和底层细节）
- 是否存在过度抽象？（为了"以后可能"而添加的抽象）
- 是否存在欠抽象？（重复代码没有合理的抽象）

### 3. 依赖方向和数据流
- 数据流向是否清晰可追踪？
- 依赖方向是否稳定？高层模块是否依赖低层细节？
- 是否存在不必要的中间状态？
- 突变（mutation）发生在哪里？是否可控？
- 数据一致性的保证是什么？

### 4. 耦合和内聚
- 高内聚：一个模块内的元素是否真正相关？
- 低耦合：模块间的依赖是否最小化？
- 是否存在"散弹式修改"（改一个功能需要改很多模块）？

### 5. 可测试性（只看架构影响）
- 架构是否支持单元测试？
- 集成测试的边界在哪里？
- 是否有难以测试的紧耦合组件？

### 6. 可演进性
- 这个架构在 6 个月 / 2 年后是否仍适用？
- 哪些设计决定会让未来改变变得困难？
- 如果有新的需求类型出现，架构能否适应？

## 输出格式

### 总体评价
架构专项健康程度（1-10）及一句话总结。

### 结构性问题
影响架构长期健康的系统性问题。

### 局部问题
可以局部修复的设计问题。

### 建议
按优先级排列的改进建议。

### 是否需要升级为综合审计
如果发现问题已经超出架构边界，说明是否建议追加 `coding-agent/independent-audit.md`。

---

## 使用提醒

- 架构评审关注的是"结构"和"关系"，不是代码风格
- 不要在此模板中展开完整测试、安全、部署审计；这些属于 `coding-agent/independent-audit.md`
- 评审结果可能有争议——聚焦在"事实"而非"偏好"
- 良好的架构评审应该让团队感觉"被揭示了真相"而非"被批评了"
