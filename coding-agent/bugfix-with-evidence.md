# Bugfix with Evidence —— 证据驱动的 Bug 修复

## 适用场景

- 有一个明确报告的 bug 需要修复
- bug 的原因尚不清楚，需要调查
- 过去尝试修过但没修好，或者修好了又复发

## 不适合场景

- bug 的原因已经明确，只需要改一行（直接改，不需要完整流程）
- 是配置问题或环境问题，不是代码问题
- 是性能问题（用专门的 profiling 流程）

# Bugfix Protocol —— Evidence First

你接到一个 bug 报告。在动手修改代码之前，按以下流程操作。

## Phase 1：收集证据（BEFORE TOUCHING CODE）

### 1.1 理解报告
- bug 的具体表现是什么？（用户看到了什么 vs 期望看到什么）
- 能否复现？复现步骤是什么？
- 影响范围？（多少用户、什么环境、什么版本）

### 1.2 检查证据链
按以下顺序检查：
1. **日志** —— 是否有相关的错误日志？
2. **运行时事件** —— 是否有监控/报警？
3. **checkpoint / persisted state** —— 是否有状态快照？
4. **observer / debug 输出** —— 是否有调试信息？
5. **session data** —— 是否有会话数据异常？
6. **stack trace** —— 是否有堆栈信息？
7. **real data flow** —— 实际的数据流向是什么？

### 1.3 形成假设
基于收集到的证据，形成关于根本原因的假设。至少要有一个假设，但不要固执于第一个假设。

## Phase 2：验证假设

1. 设计验证方法（如何证明这个假设是对的？）
2. 执行验证
3. 如果假设不成立，回到 Phase 1 收集更多证据
4. 如果假设成立，进入 Phase 3

## Phase 3：最小修复

1. 设计最小修复方案
2. 评估修复的影响范围（会不会影响其他功能？）
3. 实施修复
4. 验证修复（bug 不再复现 + 现有测试仍然通过）

## Phase 4：防止复发

1. 是否应该添加测试以防止回归？
2. 是否有同类 bug 可能存在于其他位置？
3. 修复是否揭示了一个系统性问题？（如果是，记录下来但不在此轮修复）

## 反模式（绝对不要做的事）

- 看到 bug 描述就开始改代码
- 不查日志就猜测原因
- 改了好几处还没修好就继续改更多处
- 修好后不验证就宣布完成
- 用 hard-coded bypass、string-matching hack 或临时 filter 隐藏症状
- 修好后不检查是否有同类问题

---

## 使用提醒

- Phase 1 是最重要也是最容易被跳过的步骤——强制要求先查证据
- 如果项目中无法获取日志/监控，诚实说明证据不足，基于代码分析形成假设
- 这个模板的哲学是：**找到根因比快速补丁重要**
