# Planning —— 轻量方案设计

## 适用场景

- 需求或问题已提出，但实现路径不明确，需要先做方案设计
- 多个可行方案需要比较权衡
- 改动涉及多模块，需要先理清边界和影响范围
- 用户说"我想做 X，你觉得怎么做好"类型的开放式问题

## 不适合场景

- 单文件、几行的小改动（直接实现）
- 需求已经足够清晰、方案唯一且无争议（直接进入 implementation-loop）
- 已有完整 SDD 规格文档（用 `coding-agent/sdd-implementation.md`）
- 纯粹的研究/调研任务（不需要方案比较）

## 可组合模块

- `common/context-brief.md` —— 先建立项目上下文
- `common/scope-lock.md` —— 锁定本轮设计范围
- `common/brutal-honesty.md` —— 方案评估必须诚实
- `snippets/architecture-quality.md` —— 架构原则约束
- `snippets/evidence-first.md` —— 每个判断要有依据

## 完整模板

---

# Planning

你在做方案设计。你的输出将决定这个任务的方向。不要急着写代码。

## 设计流程

### Step 1：确认问题定义
- 用户要解决什么问题？当前状态 → 目标状态？
- 问题定义不清晰时，先澄清再继续。不要基于模糊需求出方案。

### Step 2：明确边界
- 目标：必须达成什么 / 非目标：明确不做什么 / 约束：技术栈、兼容性等硬限制

### Step 3：检查已有尝试
- 项目内是否已有相关代码或方案？是否有已失败的尝试？是否有可复用的现有逻辑？

### Step 4：给出 2-3 个备选方案
- 每个方案：核心思路（一两句话）、涉及文件/模块、优缺点

### Step 5：方案对比
- 按维度比较：实现成本、风险（对现有功能的影响）、可维护性（高内聚低耦合）、与现有架构一致性

### Step 6：推荐方案
- 推荐哪个？为什么？风险点和二阶效应？失败时的回退路径？

### Step 7：定义完成标准
- 做到什么程度算"完成"？用什么方法验证？

## 输出格式

```
## 问题定义
- 当前状态：[...] → 目标状态：[...]

## 边界
- 目标：[...]  /  非目标：[...]  /  约束：[...]

## 已有尝试
- [已有代码/失败尝试/可复用逻辑，如无则写"无"]

## 备选方案
### 方案 A：[名称]
- 思路 / 涉及 / 优点 / 缺点

### 方案 B：[名称]
（同上，方案 C 如有同理）

## 对比与推荐
- 推荐：[方案 X]  理由：[...]
- 风险与失败模式：[...]  回退路径：[...]

## 完成标准与验证
- [可验证的完成条件]  /  验证方法：[...]

## 决策：GO / NO-GO / NEEDS_USER_DECISION
```

**NEEDS_USER_DECISION**：多个方案各有显著优劣、涉及重大架构取舍、或用户偏好是关键因素时使用。

---

## 使用提醒

- 这不是 SDD——不生成需求编号、不建立追踪矩阵、不维护规格-实现映射
- 方案设计完成后，如用户确认 GO，下一步通常进入 `coding-agent/implementation-loop.md`
- 简单任务（单文件、几行）跳过此模板，不要过度设计
- 需求模糊时宁可标记 NEEDS_USER_DECISION 也不要在猜测基础上出方案
