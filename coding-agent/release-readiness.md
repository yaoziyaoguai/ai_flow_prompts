# Release Readiness —— 发布就绪检查

## 适用场景

- 准备发布新版本到生产环境前
- 需要确认所有质量门已通过
- 需要生成发布检查清单
- 需要确认回滚方案就绪

## 不适合场景

- 非生产环境的部署（简化检查即可）
- 内部工具的非关键发布
- 持续部署流水线中的自动发布（流水线本身已包含检查）
- 需要实际执行发布、部署、push 或 tag 的任务（本模板只做就绪检查）

## 可组合模块

- `common/context-brief.md` —— 理解当前状态
- `common/brutal-honesty.md` —— 诚实的发布评估
- `common/red-team.md` —— 攻击发布计划
- `coding-agent/independent-audit.md` —— 发布前的最终审计（如果之前没有做过）

# Release Readiness Checklist

对即将发布的版本进行就绪检查。

## 操作边界

- 只做 release readiness review，不执行发布。
- 不部署到任何环境。
- 不运行会修改远端状态的命令。
- 不执行 `git push`、`git tag` 或修改 remote。
- 如果发现需要发布执行步骤，只输出建议和阻塞项，等待用户明确授权。

## 证据规则

- 每个检查项的判定必须基于**可验证的证据**，不得基于推测或假设。
- 如果某项检查的证据不存在或无法获取，**必须标记为 UNKNOWN**，不得标记为 PASS。
- UNKNOWN 项默认视为阻塞项，除非用户明确声明该项不适用（标记 N/A）。

## 检查维度

### 1. 代码质量门
- [ ] 所有 CRITICAL 和 HIGH 级别的审计问题已修复
  - **证据**：审计报告中的问题状态列表
  - **验证**：确认审计报告中每个 CRITICAL/HIGH 项的修复记录
- [ ] 代码评审已完成
  - **证据**：PR review 记录或代码评审报告
- [ ] 所有测试通过
  - **证据**：用户指定的测试命令输出（如 `[your-test-command]`）
  - **若无证据**：标记 UNKNOWN
- [ ] 没有已知的 release-blocking bug
  - **证据**：issue tracker 中标记为 release-blocking 的项均已关闭

### 2. 安全性
- [ ] 安全扫描通过
  - **证据**：安全扫描工具输出或项目已有的安全扫描报告
  - **若无证据**：标记 UNKNOWN
- [ ] 没有硬编码的 secret
  - **证据**：secret 扫描工具输出或人工审查结论
- [ ] 依赖漏洞检查通过
  - **证据**：依赖扫描工具输出（如 `[your-dependency-scan-command]`）
- [ ] 认证和授权路径已验证
  - **证据**：测试用例或安全审查记录

### 3. 性能
- [ ] 性能测试通过
  - **证据**：性能测试报告或 benchmark 输出
  - **若无证据**：标记 UNKNOWN
- [ ] 没有已知的性能退化
  - **证据**：与上一版本的性能对比数据
- [ ] 关键路径响应时间在可接受范围内
  - **证据**：监控数据或负载测试结果

### 4. 数据
- [ ] 数据库迁移已测试
  - **证据**：在生产数据库副本上的迁移测试结果
  - **若无证据**：标记 UNKNOWN
- [ ] 迁移有回滚方案
  - **证据**：回滚方案文档或回滚测试记录
- [ ] 数据备份已确认可用
  - **证据**：最近的备份恢复测试记录
- [ ] 没有数据丢失风险
  - **证据**：数据完整性验证结果

### 5. 部署
- [ ] 部署步骤已文档化
  - **证据**：部署 runbook 或部署文档
- [ ] 回滚方案已就绪且已测试
  - **证据**：回滚 runbook 及测试记录
- [ ] 部署所需凭证和权限已确认
  - **证据**：凭证/权限检查清单
- [ ] 第三方依赖已确认可用
  - **证据**：第三方服务状态页面或健康检查结果

### 6. 监控
- [ ] 关键指标有监控
  - **证据**：监控 dashboard 配置或截图
- [ ] 告警规则已配置
  - **证据**：告警规则配置及测试记录
- [ ] On-call 人员已确认
  - **证据**：On-call 排班表或确认记录
- [ ] 日志收集正常工作
  - **证据**：最近的日志查询结果

### 7. 文档
- [ ] Release notes 已准备
  - **证据**：CHANGELOG 或 release notes 文件
- [ ] Breaking changes 已标记
  - **证据**：release notes 中的 breaking changes 章节
- [ ] 已知问题和限制已记录
  - **证据**：KNOWN_ISSUES 文件或 release notes 中的已知问题章节
- [ ] API 文档已更新（如有变更）
  - **证据**：API 文档 diff 或更新记录

### 8. 版本与 Git 状态
- [ ] 版本号已更新
  - **证据**：version file / CHANGELOG / `[your-version-file]` 中的版本号
- [ ] 分支状态清晰
  - **证据**：`git status` 输出——无未提交变更，分支与 remote 关系明确
- [ ] Tag 准备就绪（如使用 tag 发布）
  - **证据**：确认 tag 命名规范、tag message 已准备
- [ ] 与 remote 的 ahead/behind 状态已确认
  - **证据**：`git branch -v` 或等效输出

## 输出

### 发布判定
- GO / NO-GO / CONDITIONAL GO（需满足条件 X 后发布）
- 任何 UNKNOWN 项默认导致 CONDITIONAL GO 或 NO-GO

### 风险清单
列出发布后需要关注的 top 3 风险。

### 回滚触发条件
明确什么情况下应该回滚。

---

## 使用提醒

- 每一项都需要**可验证的证据**——没有证据就标记 UNKNOWN，不要推测通过
- 这是 release readiness review prompt，不是发布执行脚本
- 如果某项不适用（如无数据库迁移），标记为 N/A 而非跳过
- NO-GO 判定很正常——宁可延迟发布，不要带病上线
- CONDITIONAL GO 的条件必须在发布前确认完成
