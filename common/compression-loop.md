# Compression Loop —— 长上下文压缩

## 适用场景

- 对话已经变得很长（超过正常上下文窗口的 60%），需要压缩
- 需要在压缩后保持关键信息和决策链不丢失
- 需要为下一阶段的对话清理"上下文债务"
- 项目持续多轮开发，每轮之间需要压缩交接

## 不适合场景

- 对话还很短，不需要压缩
- 即将结束会话，不需要保留上下文
- 信息是纯线性的（如读文档），压缩无意义

## 可组合模块

- `common/context-brief.md` —— 压缩后重新建立上下文地图
- `common/scope-lock.md` —— 压缩后重新锁定范围
- `common/assumption-audit.md` —— 审计压缩过程中可能丢失的关键信息

## 完整模板

---

# Compression Loop Protocol

对话上下文已经变长。现在执行压缩循环。

## 压缩目标

将当前的对话上下文压缩为一份**结构化交接文档**，保留一切后续工作必需的信息，丢弃冗余。

## 压缩维度

### 1. 项目状态快照

- 当前分支、最近 commit
- 工作目录状态（修改了哪些文件）
- 运行的进程 / 服务状态

### 2. 已完成事项

- 本轮完成了什么（按时间顺序或按模块）
- 每个完成事项的关键决策和理由
- 哪些尝试过但被放弃，为什么

### 3. 当前进行中

- 正在做什么
- 进行到哪一步了
- 遇到了什么障碍

### 4. 待办事项

- 还需要做什么
- 优先级排序
- 是否有阻塞依赖

### 5. 关键决策记录

对本轮做过的每一个不可逆决策：
- 决策内容
- 为什么这样决定
- 有哪些替代方案被拒绝了
- 如果将来要改变这个决策，需要考虑什么

### 6. 已知风险和未解决的问题

- 当前已知的问题（技术债务、不完美的实现）
- 需要在后续关注的信号（性能、稳定性、安全）
- 未解决的 open question

### 7. 关键文件清单

列出本轮涉及的关键文件及其角色：
```
path/to/file1 — [角色/用途]
path/to/file2 — [角色/用途]
```

## 压缩后

压缩后的文档应满足：
- 一个新加入的开发者读完就能继续工作
- 明显短于原始上下文，只保留后续工作必需的信息；如果必须保留大量细节，按“关键决策 / 当前状态 / 待办 / 风险”分区
- 关键决策和理由全部保留

---

## 使用提醒

- 压缩是一个"有损"过程——确保关键决策和理由不丢失是最高优先级
- 压缩后，在新的会话中加载压缩文档作为 context brief
- 压缩频率建议：每完成一个独立阶段或对话达到 context 窗口 60% 时
- 压缩不是"总结"——重点在于保留后续工作所需的结构化信息
