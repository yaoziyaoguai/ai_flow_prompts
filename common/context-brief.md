# Context Brief —— 上下文地图建立

## 适用场景

- 开始任何任务前，建立项目上下文地图
- AI 对项目结构、技术栈、关键约束还不清楚时
- 需要在多轮对话中保持一致上下文时
- 新加入的 Agent 需要快速理解项目

## 不适合场景

- 已经建立了清晰的上下文，无需重新建立
- 任务非常简单，不需要项目级上下文
- 项目还处于早期探索阶段，结构未定型

## 可组合模块

通常作为第一个加载的模板，与以下组合：
- `common/scope-lock.md` —— 建立上下文后锁定范围
- `common/assumption-audit.md` —— 用于验证上下文是否准确
- 任何 `coding-agent/` 模板 —— 在执行前先建立上下文

## 完整模板

~~~text
# Context Brief

在开始执行任何任务之前，先建立当前上下文的清晰地图。**先理解，再行动——不要直接跳入建议或实现。**

## 发现步骤（按顺序执行）

### Step 1：环境快照

运行只读命令获取当前状态：
- `pwd` — 确认工作路径
- `git status --short` — 变更状态
- `git log --oneline --decorate -5` — 最近提交

如果路径不符预期、或存在 merge conflict / rebase in progress / detached HEAD，**停止并报告**。

### Step 2：规则文件发现

检查是否存在以下文件并读取关键规则：
- `AGENTS.md` — 项目级 Agent 行为约束
- `CLAUDE.md` — Claude Code 特定约束
- `README.md` — 项目概述和快速入门

只提取与本轮任务相关的规则和约束。不逐字复制全文。

### Step 3：项目结构扫描

- 顶层目录和关键子目录
- 技术栈线索（`package.json` / `pyproject.toml` / `go.mod` / `Makefile` 等）
- 测试目录和测试命令

### Step 4：敏感文件识别（不读取内容）

扫描 `.env` / `*.key` / `credentials.*` / `secrets.*` / `logs/` / `sessions/` / `runs/` / `data/`。**只记录存在性**。如果发现，报告中标记 `[敏感文件，未读取]`。

### Step 5：确认任务边界

从用户描述或 `current-task.md` 中提取：
- 本轮目标和成功标准
- **明确不做什么**（如果用户未声明，基于上下文推断并请用户确认）
- 已有尝试和已知卡点

## 输出格式

按以下结构输出，不确定的项标记 `UNKNOWN`，不要猜测：

```
## 我理解的项目背景
- 项目：[名称/定位]
- 技术栈：[语言/框架]
- 当前分支：[分支名]，[N] 个未提交文件

## 关键约束
- [约束 1：来源 AGENTS.md]
- [约束 2：用户声明]
- …

## 本轮任务
- 目标：[…]
- 不做什么：[…]
- 成功标准：[…]

## 不确定点（UNKNOWN）
- [信息缺失项 1] — 需要用户确认
- [信息缺失项 2] — 不影响继续但标记追踪

## 建议下一步
- [1-3 条建议，基于已确认的信息]
```

**如果 UNKNOWN 项阻塞了任务执行，先确认再继续。不阻塞的 UNKNOWN 可以边做边澄清。**
~~~

## 使用提醒

- 如果项目已有 AGENTS.md 且上下文清晰，使用简化版（只确认"变化了什么"）
- 持续进行的会话不需要重新完整执行——只更新变化部分
- 不确定的项用 UNKNOWN 标记，不要猜测填充
