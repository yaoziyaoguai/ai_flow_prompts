# Premortem —— 项目前期验尸

## 适用场景

- 项目启动前，识别可能失败的原因
- 重大里程碑或发布前
- 技术方案确定后、开始实现前
- 新团队成员加入时，用来对齐风险认知
- 时间紧、资源少的项目——提前暴露风险

## 不适合场景

- 项目已经上线运行（应该用 postmortem）
- 方案还在剧烈变动中，过早验尸没有意义
- 风险已知且已充分讨论，不需要仪式化的流程

## 可组合模块

- `common/brutal-honesty.md` —— 验尸必须诚实
- `common/red-team.md` —— 红队 + 验尸是强力组合
- `common/assumption-audit.md` —— 检查导致失败的假设
- `coding-agent/independent-audit.md` —— 技术层面的验尸审计

## 完整模板

# Premortem Protocol

现在执行前期验尸。假设项目已经完成，但**结果是一场灾难**。我们需要找出为什么会失败。

## 场景设定

时间：项目计划完成日期 + 1 周后
状态：项目宣告失败
问题：失败的根本原因是什么？

## 分析维度

### 1. 技术失败

- 什么技术决策导致了失败？
- 哪些技术风险被低估了？
- 性能、可扩展性、安全性方面哪里出了问题？
- 有什么技术债务在中期就拖垮了进度？

### 2. 流程失败

- 需求在什么时候被严重误解？
- 沟通在哪个环节断裂了？
- 什么关键的检查点被跳过了？
- 外部依赖在什么地方让人失望了？

### 3. 协作和知识传递失败

- 哪些项目知识只存在于对话或个人记忆中，没有沉淀到文档？
- Coding Agent 对任务边界、成功标准或项目规则的理解可能在哪里断裂？
- 哪些评审、交接或确认环节缺失，会导致错误被带入实现？

### 4. 范围失败

- 范围在什么时候失控了？（scope creep）
- 什么"必须做"的功能其实根本不必要？
- 什么被过度设计了？

### 5. 外部失败

- 市场、监管、竞品或其他外部因素如何影响了项目？
- 什么外部假设不成立了？

## 输出

对每个识别出的失败原因：
1. 描述失败的具体场景
2. 评估发生的可能性（高/中/低）
3. 评估如果发生的影响程度
4. 提出现在的预防措施

最后给出：**如果只能预防 3 个失败原因，应该选哪 3 个？**

---

## 使用提醒

- 验尸不是否定项目——所有项目都有失败的可能，提前识别让项目更强大
- 验尸结果应该转化为具体的预防措施，而不是留在文档里积灰
- 如果验尸发现了 CRITICAL 级别且无法预防的风险，需要认真考虑是否调整方向
- 这个模板最有效的使用时机是：方案已经形成但还未投入大量资源执行
