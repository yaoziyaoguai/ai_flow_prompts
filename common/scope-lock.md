# Scope Lock —— 范围锁定

## 适用场景

- 容易发散的任务——修 bug 时顺手重构、加功能时顺带优化
- Coding Agent 倾向于"多做一点"的场景
- 需要明确界定"做什么"和"不做什么"的任务
- 多人协作时防止任务范围重叠冲突

## 不适合场景

- 探索性任务，范围确实不明确
- 需要 Agent 主动发现问题并提出建议的场景
- 代码简单到不可能发散的情况

## 可组合模块

- `common/context-brief.md` —— 先建立上下文，再锁定范围
- 几乎所有 `coding-agent/` 模板都应该加载此模板
- `snippets/no-push-no-tag.md` —— 范围锁定的常见补充

## 完整模板

```text
# Scope Lock

本次任务的范围已锁定。请严格遵守以下边界。

## 本轮目标

[在此描述本轮要完成的具体目标]

## 允许的操作

- [列出明确允许的操作]

## 禁止的操作

- 不要修改与目标无关的文件
- 不要重构不相关的代码
- 不要添加"可能以后有用"的功能
- 不要修改测试（除非目标需要更新测试）
- 不要做"顺手"的优化
- 不要引入新的依赖，除非确有必要
- 不要扩大范围——如果一个改动引发了连锁反应，先停下来报告
- 不要 push / tag / 修改 remote 配置

## 遇到边界模糊时的处理

如果你发现：要完成目标 A，似乎必须先做 B，而 B 在禁止范围内——

**停止并报告**：
1. 描述 A 和 B 之间的关系
2. 提出选项（放宽范围做 B / 缩减 A 的范围 / 找替代方案）
3. 等待确认后再继续

## 完成后

- 确认只修改了允许操作范围内的文件
- 列出实际修改的文件清单
- 如果做了什么额外操作，必须解释原因
```

## 使用提醒

- "允许的操作"列表要具体，不要写"修 bug"这种模糊条目
- "禁止的操作"列表是关键——宁可多写几条，Agent 一般不会因为限制太多而出问题
- 如果任务确实需要灵活范围，不要用此模板
