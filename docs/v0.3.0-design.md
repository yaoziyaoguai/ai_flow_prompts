# v0.3.0 SDD Compliance / Lightweight RTM Design

> **状态**：设计文档，不代表已经决定实现。实现前必须经过独立红队审计。
> **范围**：v0.3.0 主线收缩为 SDD Compliance / Lightweight RTM。Project Task Ledger 降级为 opt-in 候选，不进入 v0.3.0 默认实现范围。

---

## 1. 背景与目标

ai_flow_prompts v0.2.0 已支持 Workflow Router。但在 SDD 实现任务中，存在一个未被解决的诚实问题：

- Agent 容易声称"已完成"，但没有逐项对照 SDD requirement
- Agent 容易把 PARTIAL 包装成 FULL
- Agent 容易把 UNKNOWN 当 PASS
- 用户无法清楚知道哪些 SDD 要求未完成、未验证、或偏离

**v0.3.0 目标**：为 SDD 实现任务提供轻量 Requirements Traceability Matrix（RTM），强制 Agent 诚实披露每个 requirement 的 **Implementation Evidence**、**Verification Evidence**、**Verification Status** 和 **Compliance**。

**不在 v0.3.0 范围**：通用 Project Task Ledger（每个用户请求都记录）。该机制降级为 opt-in / future candidate，需真实项目试用 v0.2.0 后再评估。

---

## 2. 解决什么问题

| 问题 | 当前状态 | v0.3.0 方案 |
|---|---|---|
| SDD requirement 缺少 **Implementation Evidence** | Agent 口头声称完成，无逐项对照 | 每个 requirement 必须标注代码位置 |
| 缺少 **Verification Evidence** | 测试可能未运行，或结果未关联到 requirement | 每个 requirement 必须标注验证证据 |
| 缺少 **Verification Status** | "完成"含义模糊（写完代码？测试通过？手动验证？） | 引入 VERIFIED / NOT_RUN / FAILED / BLOCKED / UNKNOWN |
| PARTIAL 包装成 FULL | Agent 倾向于展示"都做好了" | 诚实规则：PARTIAL ≠ FULL，UNKNOWN ≠ PASS |
| 多任务 SDD 实现无法追溯 | 每次实现是独立会话，无跨任务对照 | SDD Compliance Report 按 Task Reference 关联 |

---

## 3. 与 v0.2.0 Workflow Router 的关系

```
用户自然语言意图
        ↓
Workflow Router（选择 sdd-implementation.md）
        ↓
current-task.md（定义本轮范围）
        ↓
Coding Agent 执行 SDD 实现
        ↓
SDD Compliance Report（收尾产物，仅 SDD 任务）
```

Router 不新增"SDD compliance"作为独立意图——它不是独立工作流，而是 `sdd-implementation.md` 的收尾产物。

## 5. Project Task Ledger：暂缓 / opt-in 候选

Project Task Ledger 的核心思想（跨会话项目任务记忆）有价值，但红队审计指出以下风险：

| 风险 | 说明 |
|---|---|
| **文档税** | 每个小任务都写一行记录会积累维护成本 |
| **虚假项目记忆** | Agent 写的摘要可能不准确，积累成误导 |
| **仓库污染** | `docs/agent/` 下文件持续增长 |
| **merge 冲突** | 多分支同时维护 ledger 会产生冲突 |
| **重复信息** | 与 commit diff、PR 描述、implementation-notes 重叠 |
| **敏感信息污染** | User Request / Summary / Known Issues 字段可能诱导记录 secret 或业务数据 |
| **T001 不稳定** | 在分支、merge 场景下 ID 递增不可靠 |
| **Markdown 当数据库** | Markdown 表格不适合机器维护的事实来源 |

**结论**：Project Task Ledger **不默认启用、不要求每个请求分配 Task ID、不进入 v0.3.0 最小范围**。保留为 opt-in 候选，适合长期 Agent 项目、SDD 项目、审计修复链路等场景，需真实项目验证后再设计。

---

## 6. Task Reference（替代 Task ID）

SDD Compliance Report 需要关联到具体任务，但不强制 T001 递增 ID。Task Reference 可以来自：

- current-task.md 标题
- issue ID / PR ID
- branch name
- 用户提供的任务标签
- 可选的本地 task ID（如项目未来自行维护）

v0.3.0 不为 Task Reference 设计强制格式。SDD Compliance Report 模板中使用 `[Task Reference]` 占位符。

## 7. SDD Compliance Report：轻量 RTM

### 文件位置

`[project-path]/docs/agent/sdd-compliance/[task-reference].md`

> **注意**：`docs/agent/sdd-compliance/` 是**被接入的目标项目**的可选输出路径示例，不是 ai_flow_prompts 本仓库要创建的目录。v0.3.0 只提供模板和规则，不创建目标项目运行产物目录。

### Compliance Matrix 字段

| 字段 | 说明 | 示例 |
|---|---|---|
| **Requirement ID** | SDD 原文编号；如 SDD 无编号，使用 `AUTO-R001`（仅为本次报告的临时编号，auto-generated, not from original SDD，不得视为原始 SDD 的稳定 **Requirement ID**） | `SPEC-03` |
| **Requirement Summary** | 脱敏摘要，不复制敏感原文 | `"登录失败 3 次后锁定账户"` |
| **Implementation Evidence** | repo 相对路径 / 代码位置 / PR ID / commit hash，不贴完整代码，不写内部 URL | `src/auth/login.ts:45-67` |
| **Verification Evidence** | 测试文件、测试命令、手动验证、静态检查、审计结果等验证来源 | `tests/auth/test_login.py::test_lockout` |
| **Verification Status** | VERIFIED / NOT_RUN / FAILED / BLOCKED / UNKNOWN | `VERIFIED` |
| **Compliance** | FULL / PARTIAL / NOT_DONE / UNKNOWN / N/A | `FULL` |
| **Deviation / Gap** | 如非 FULL，如实说明偏离 | `"锁定时长 10 分钟而非 15 分钟——测试环境限制"` |
| **Follow-up** | 是否需要用户确认或后续补救 | `Yes → 确认锁定时长规格` |

### 状态定义与硬性规则

**Compliance 状态**：

| 状态 | 定义 | 硬性规则 |
|---|---|---|
| **FULL** | 完全按规格实现，已运行验证并通过 | **必须同时有 **Implementation Evidence** 和 **Verification Evidence**，且 **Verification Status** = VERIFIED**。FULL 是最严格状态，不是默认状态 |
| **PARTIAL** | 部分实现、部分验证或存在明确 gap | **禁止包装成 FULL**——偏离必须写在 **Deviation / Gap** 列 |
| **NOT_DONE** | 未实现，或没有可接受的 **Implementation Evidence** | **禁止沉默跳过**——必须写明原因 |
| **UNKNOWN** | 当前证据不足，无法判断是否实现或是否满足要求 | **UNKNOWN 不能当 PASS**——无法确认就是 UNKNOWN，必须标注 |
| **N/A** | 经用户或规格确认不适用于本次任务 | 必须说明理由，不强制填写 evidence |

**Verification Status**：

| 状态 | 定义 |
|---|---|
| **VERIFIED** | 测试已运行并通过 |
| **NOT_RUN** | 测试存在但本轮未运行 |
| **FAILED** | 测试运行但未通过 |
| **BLOCKED** | 无法验证（依赖缺失、环境问题等） |
| **UNKNOWN** | 无法确认验证状态 |

**硬性规则**：
- **FULL 必须同时有 **Implementation Evidence** 和 **Verification Evidence**，且 **Verification Status** = VERIFIED**——只写了代码但没运行验证，不能标 FULL；缺少任一 evidence，不能标 FULL
- **没有 **Verification Evidence** 不能标 FULL**——如果 requirement 不需要自动化测试，必须有其他验证证据（手动检查、静态分析、审计等）并记录在 **Verification Evidence** 中；否则标 PARTIAL 或 UNKNOWN
- **FULL 是最严格状态，不是默认状态**——默认从 UNKNOWN 或 PARTIAL 开始，积累证据后升级
- **UNKNOWN 不能当 PASS**——无法确认就是 UNKNOWN，必须标注
- **PARTIAL 不能包装成 FULL**——有偏离就说
- **NOT_DONE 不能沉默**——没做就是没做
- 如果 SDD 无 requirement ID，使用 `AUTO-R001`（仅为本次报告的临时编号，auto-generated, not from original SDD，不得视为原始 SDD 的稳定 **Requirement ID**）

### 示例

```markdown
# SDD Compliance Report —— [Task Reference]

- **规格文档**：`docs/spec/user-auth.md`
- **Task Reference**：T003（或 issue #42 / branch feat/oauth）
- **日期**：2026-05-19

## Compliance Matrix

| Req ID | Requirement Summary | Implementation Evidence | Verification Evidence | Verification Status | Compliance | Deviation / Gap | Follow-up |
|---|---|---|---|---|---|---|---|
| SPEC-01 | 登录失败 3 次锁定 15 分钟 | `src/auth/login.ts:45-67` | `tests/auth/test_login.py::test_lockout` | VERIFIED | PARTIAL | 锁定时长 10 分钟——测试环境限制 | Yes |
| SPEC-02 | 密码 ≥8 位含特殊字符 | `src/auth/validate.ts:12-30` | `tests/auth/test_validate.py::test_password` | VERIFIED | FULL | — | No |
| SPEC-03 | OAuth2 集成 | — | — | BLOCKED | NOT_DONE | 第三方依赖未就绪 | Yes |
| SPEC-04 | 会话超时 30 分钟 | `src/session.ts:20` | — | NOT_RUN | PARTIAL | **Implementation Evidence** 存在，**Verification Evidence** 缺失；**Verification Status** = NOT_RUN | Yes |
| SPEC-05 | 审计日志 | — | — | UNKNOWN | UNKNOWN | 未找到对应实现或测试 | Yes |

## 诚实声明

- SPEC-01 标为 PARTIAL：锁定时长偏离，已如实标注
- SPEC-04 标为 PARTIAL 而非 FULL：**Verification Evidence** 缺失，**Verification Status** = NOT_RUN
- SPEC-05 标为 UNKNOWN：未假设通过
- SPEC-03 标为 NOT_DONE：已说明原因
```

## 8. 安全边界

### 禁止记录

- secret / API key / token / 密码
- .env 内容
- cookie / session token
- 内部 URL（GitHub / GitLab / internal Git 等完整链接，含 query 参数）
- 真实日志原文
- 用户 PII / 客户数据
- 完整业务数据
- 内部敏感原文

### 允许记录

- repo 相对路径（`src/auth/login.ts:45`）
- 文件名
- 测试命令（`pytest tests/auth/ -k test_lockout`）
- 验证结论（`12/12 PASS`）
- 脱敏摘要（不包含具体数据值）
- issue ID / PR ID / commit hash（仅标识符，不写完整 URL）
- requirement ID

### 硬性规则

- Evidence 字段只能写路径、命令、脱敏摘要、ID 标识符
- 如果必须引用内部系统，只能写脱敏标识或 ID（如 PR ID、commit hash），不写完整 URL
- 如果 requirement 本身包含敏感内容，摘要必须脱敏
- 涉及敏感数据的任务在 Compliance Report 中标注 `[sensitive — details not recorded]`

## 9. Workflow 接入建议

| 模板 | 是否接入 | 原因 |
|---|---|---|
| `sdd-implementation.md` | **必须接入** | v0.3.0 唯一主线——SDD 实现完成后输出 Compliance Report |
| `implementation-loop.md` | **可选接入** | 仅当用户明确要求 SDD 式合规对照时使用 |
| `commit-readiness.md` | **可选接入** | 可检查 SDD Compliance Report 是否存在，但不生成报告 |
| 其他所有 workflow | **不建议 v0.3.0 接入** | 避免污染 Router/current-task 边界，避免文档税 |

Router 不识别"更新 compliance report"作为独立工作流意图。SDD Compliance 是 `sdd-implementation.md` 的收尾产物，不是通用任务路由目标。

---

## 10. 负担控制

- 非 SDD 项目不启用 SDD Compliance Report
- 只有 SDD 实现任务强制输出 Compliance Report
- 小任务（非 SDD、单文件改动）不生成任何额外文档
- SDD Compliance Report 不重复 commit diff、PR 描述、implementation-notes——只做 requirement 对照
- 不为记录而记录

## 12. 风险与反对意见

| 风险 | 严重度 | 如何通过收缩缓解 |
|---|---|---|
| **Agent 把 PARTIAL 包装成 FULL** | HIGH | 拆分 **Implementation Evidence** / **Verification Evidence** / **Verification Status**——缺少任一维度就不能标 FULL |
| **Evidence 字段泄露敏感信息** | HIGH | 8 条禁止记录 + 3 条允许记录 + 脱敏标注规则 |
| **SDD 无 requirement ID 导致对照形同虚设** | MEDIUM | 允许 AUTO-R001 但标记不稳定，建议用户先编号 |
| **SDD Compliance Report 重复 implementation-notes** | MEDIUM | Compliance Report 是 requirement 对照表，不是实现过程记录——两者不重叠 |
| **非 SDD 项目被强制接入** | LOW | 只有 sdd-implementation.md 强制输出，其他 workflow 不接入 |
| **Markdown 表格不稳定** | LOW | 3 文件范围，不引入数据库或结构化存储 |
