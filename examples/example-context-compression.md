# 示例：长上下文压缩

## 场景

一个 Coding Agent 会话已经进行了 20+ 轮，完成了用户管理模块的 CRUD 和权限系统。现在需要将会话上下文压缩，准备进入下一阶段（审计和修复）。

## 组装方式

组合以下模板：
- `common/context-brief.md`
- `common/compression-loop.md`

## 组装后的完整 prompt

---

# 上下文压缩

对话已进行多轮。请在继续下一阶段之前，执行上下文压缩。

## Compression Loop

压缩目标：将当前对话上下文压缩为一份结构化交接文档。

### 请覆盖以下维度：

#### 1. 项目状态快照
- 分支、commit、文件修改状态
- 当前项目目录结构（与任务相关的部分）
- 任何运行的进程或服务

#### 2. 已完成事项
- 用户管理 CRUD（创建、读取、更新、删除）
- 权限系统（角色定义、权限分配、中间件验证）

对每个完成事项，记录：
- 关键决策和理由
- 尝试过但放弃的方案（如果有）

#### 3. 当前进行中
- 是否有未完成的事项？
- 进行到哪一步了？

#### 4. 待办事项
- 下一阶段：审计和修复
- 审计报告路径：[待定]
- 修复计划：[待定]

#### 5. 关键决策记录
记录本阶段做过的所有不可逆决策：
- 决策内容
- 为什么这样决定
- 替代方案及其被拒绝的原因

#### 6. 已知风险和未解决问题
- 权限中间件的性能尚未在负载下测试
- 用户删除操作未实现软删除（当前是硬删除）
- 缺少集成测试

#### 7. 关键文件清单
列出所有涉及的文件及其角色。

### 压缩质量要求
- 一个新加入的开发者读完压缩文档后应该能继续工作
- 保留所有关键决策和理由
- 不要遗漏已知问题和风险

---

## 使用说明

1. 这是一个示例——实际使用时，需要 Agent 根据真实的对话内容来填充
2. "已完成事项"和"关键决策记录"是最容易遗漏的——确保压缩时保留
3. 压缩后的文档保存在项目 `docs/prompts/history/` 中，供后续会话和开发者参考
4. 建议在 context 窗口使用率达到 60-70% 时执行压缩

## 预期产物

一份 500-1500 字的结构化交接文档，包含项目状态、已完成事项、决策记录、风险和待办。

## 坏产出信号

- 只有文件清单，没有决策记录和理由——新接手的人不知道为什么当时那样做
- 遗漏了已知风险或未解决问题——这些是最容易在交接中丢失的信息
- 过度压缩导致关键上下文丢失（如"放弃了方案 A"但没说为什么）
- 压缩文档太长（>2000 字），失去了压缩的意义
