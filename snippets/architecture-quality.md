# Architecture Quality —— 架构质量约束片段

## 适用场景

- 嵌入到所有涉及代码设计和实现的模板中
- 编码规范已包含架构原则，但又需要强调时
- 审计和评审类模板中作为质量标准

## 完整片段

```text
## 架构质量约束

### 核心原则

**高内聚、低耦合**
- 一个模块内的元素应该紧密相关（高内聚）
- 模块间的依赖应该最小化（低耦合）
- 理想状态：修改一个功能只需要改一个模块

**清晰的架构边界**
- 每个模块有明确的职责范围
- 跨边界的交互通过明确的接口
- 不存在"谁都能调谁"的随意依赖

**不可变性优先**
- 创建新对象，不要修改已有对象
- 数据的变异点应该在可控的少数位置
- 避免共享可变状态

### 检查清单

- [ ] 是否新增了不必要的抽象？
- [ ] 是否破坏了现有的模块边界？
- [ ] 是否引入了循环依赖？
- [ ] 是否添加了与当前模块职责无关的代码？
- [ ] 数据流向是否清晰？
- [ ] 变异（mutation）发生的位置是否明确且可控？

### 函数和文件约束

- 函数职责单一（做一件事，把它做好）
- 文件聚焦于一个主题或领域
- 命名能揭示意图——读代码的人不需要猜测

### 编程的艺术

不是"高性能"才是好代码。好代码的特征：
- 阅读代码时，控制流清晰可追踪
- 边界明确——你知道在哪里能找到什么
- 修改一个行为时，改动集中在少数位置
- 命名一致，风格统一
- 没有"surprise"——行为符合命名和位置所暗示的
```

## 使用提醒

- 这是一个**可嵌入片段**，应在实现类和审计类模板中嵌入
- 这不是完整的编码规范——编码规范应在项目的 AGENTS.md 中定义
- "编程的艺术"部分是一种哲学提醒，可根据项目文化决定是否保留
