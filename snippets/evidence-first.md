## 证据优先原则

在修改任何代码之前，必须先收集证据。

### 证据链（按优先级）

1. **日志** —— 检查相关日志，理解当前行为
2. **运行时事件** —— 检查监控、报警、状态变化
3. **checkpoint / persisted state** —— 检查持久化状态
4. **observer / debug 输出** —— 检查调试信息
5. **session data** —— 检查会话数据
6. **stack trace** —— 检查调用栈
7. **real data flow** —— 追踪实际数据流

### 操作要求

- 在确定根因之前，不要修改代码
- 每个结论必须有文件路径、行号、日志输出等具体证据支持
- 如果证据不足，明确指出"证据不足，以下分析基于代码阅读"
- 不要因为"看起来像"而修改代码

### 反模式

- 看到一个错误信息就直接改对应代码
- 不检查调用方就修改被调用方
- 在多个可能的原因之间猜，每个都试着改一下
- 用 hard-coded string match 或 temporary filter 掩盖问题
